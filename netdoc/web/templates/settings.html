        Host discovery
      

    


    {# 2. Multi-method host discovery #}
    

      

        

          

            
            Multi-method host discovery
            PLANNED
            nmap -PE -PP -PS80,443 -PA80 -PU161
          

          

            Standard discovery relies solely on ICMP responses (classic "ping"). The problem is
            that many devices — IP cameras, Hikvision/Dahua cameras, Siemens PLCs, HP printers —
            block ICMP in firmware or via firewall rules. Such devices "look"
            as if they don't exist, even though they are working normally.
          

          

            Multi-method discovery tries several techniques at once:
          

          
    
            
  • ICMP Echo (-PE) — classic ping
    • 
            
  • ICMP Timestamp (-PP) — alternative ICMP type, bypasses some firewalls
    • 
            
  • TCP SYN ping (-PS22,80,443) — "knocks" on popular TCP ports
    • 
            
  • TCP ACK ping (-PA80) — looks like a response to an existing connection, hard to filter out
    • 
            
  • UDP ping (-PU161) — for devices with SNMP (PLCs, switches)
    • 
          

          

            
            Risk: more network traffic, longer scan time. Some devices may
            react unpredictably to TCP SYN on closed ports (rare).
          

          

            
            Feature in preparation. May detect 15–30% more devices on networks with filtered ICMP.
          

        

        

          
        

      

    


    {# 3. Randomize hosts #}
    

      

        

          

            
            Randomize host scan order
            PLANNED
            nmap --randomize-hosts
          

          

            By default nmap scans hosts sequentially (192.168.1.1, .2, .3…). On large networks this can
            mean that for tens of seconds it "attacks" only one switch or one network segment,
            which may overload it or trigger IDS alarms. Random order distributes traffic evenly.
          

          

            Particularly useful for /16 and larger networks, where sequential scanning loads
            each of dozens of switches one after another.
          

          

            
            Feature in preparation. No risks, recommended for networks larger than /24.
          

        

        

          
        

      

    


    {# ── SEKCJA: Omijanie filtrów i firewalli ─────────────────────────────── #}
    

      

        Bypassing network filters
      

    


    {# 4. Source port spoofing #}
    

      

        

          

            
            Source port spoofing
            PLANNED
            nmap -g 53
          

          

            Firewall rules often work in both directions: "block everything, but allow traffic from/to
            port 53 (DNS)". By sending scanning packets from port 53 as the source, the scanner "pretends"
            to be a DNS query — simple firewalls treat such traffic as permitted and let through
            the response from the target host.
          

          

            Example "trusted" ports that can be specified: 53 (DNS), 80 (HTTP),
            443 (HTTPS), 20 (FTP data).
          

          

            
            Risk: may cause conflicts with services running on those ports
            (e.g. local DNS server on port 53). Use only when standard scanning yields no results.
          

          

            
            Feature in preparation. Helps discover hosts/ports behind simple ACL rules.
          

        

        

          
        

      

    


    {# 5. Fragmentacja pakietów #}
    

      

        

          

            
            IP packet fragmentation
            PLANNED
            nmap -f  
            nmap --mtu 16
          

          

            TCP packets are split into small fragments (default 8 bytes) before sending.
            Simple Intrusion Detection Systems (IDS) and firewalls with packet inspection
            analyze each packet individually — they do not reassemble fragments before analysis.
            This allows fragmented scanning to "slip through" filters that would normally
            block SYN packets.
          

          

            
            High risk for industrial devices: PLCs, industrial converters
            and old switches may not support IP fragment reassembly — they receive incomplete packets
            and may freeze or restart. Use only on office IT networks, never on OT/SCADA.
          

          

            
            Feature in preparation. Disabled by default — requires deliberate enabling.

    {# 6. Bad checksum #}
    

      

        

          

            
            Checksum test (Bad Checksum)
            PLANNED
            nmap --badsum
          

          

            Sends TCP/UDP packets with a deliberately incorrect checksum. Every correctly
            implemented network stack should simply ignore them — if so, no response means a "real" host.
            However, if the host or firewall responds,
            it indicates a serious vulnerability: the device does not verify packet integrity, which can
            be exploited by attackers.
          

          

            The test result is recorded as a new vulnerability type: firewall_no_checksum_validation.
            This is particularly important for NERC CIP and IEC 62443 regulations in OT environments.
          

          

            
            Feature in preparation. Safe — causes no side effects on devices.

    {# 7. Decoy Scan — PRO #}
    

      

        

          

            
            Decoy scan
            PRO
            nmap -D RND:5,ME
          

          

            Every outgoing scan packet is accompanied by copies with fake IP addresses (decoys).
            The monitoring system or firewall sees a dozen "scanners" at once — it is hard
            to tell which is real. Useful during authorized penetration tests of your own
            infrastructure when you want to verify the effectiveness of the intrusion detection system (IDS/SIEM) —
            whether it can isolate the real source from the noise.
          

          

            
            Requires authorization: this technique may trigger security alerts and
            should only be used within a documented penetration test. Fake IPs
            must be reachable on the network, otherwise the technique is ineffective.

    {# 8. MAC Address Spoofing — PRO #}
    

      

        

          

            
            MAC address spoofing
            PRO
            nmap --spoof-mac Cisco
          

          

            The scanner sends packets impersonating the MAC address of another vendor or a specific
            device. Used when the network environment applies MAC-based access control
            (802.1X, MAC-based ACL) — the scanner "pretends" to be an approved device and gains access to
            network segments inaccessible with a standard scan.
          

          

            Available modes: impersonate a vendor (e.g. Cisco, Dell),
            random MAC (0) or a specific address (AA:BB:CC:DD:EE:FF).
          

          

            
            Risk: may disrupt switch ARP tables and cause
            address conflicts. Use only in test environments or with full authorization.
            Ineffective when the switch uses Port Security (blocks unknown MACs).

    {# 9. Zombie / Idle Scan — PRO #}
    

      

        

          

            
            Zombie host scan (Idle/Zombie Scan)
            PRO
            nmap -sI [zombie] [cel]
          

          

            The most advanced and "silent" scanning technique. The scanner never sends packets
            directly to the target — instead it uses a third host ("zombie") as an intermediary.
            As a result, only the zombie's IP address appears in the target's firewall logs, not the real
            scanner's address. The technique requires finding a host with low network traffic and a predictable
            IP sequence number (IPID).
          

          

            Use in NetDoc: verifying what a "neutral" host sees on the network — whether the firewall blocks
            traffic only from known scanner IPs, or actually filters ports regardless of source.
          

          

            
            Authorized penetration tests only: this technique is considered
            an attack method in many jurisdictions — requires written consent from the network owner. The zombie host
            must be your own device or one covered by the audit scope.
